Reflection Questions
1. Why is conversation history necessary?

Conversation history is necessary because large language models are stateless by default.
They do not remember previous messages unless the full conversation is sent again with each request.
By storing and resending prior messages:

The model understands context.
It can reference earlier questions.
It maintains logical continuity.
It supports follow-up questions.
It enables multi-turn dialogue.

Without conversation history, every message would be treated as a completely new and unrelated request.


2. What happens if you remove the system prompt?

The system prompt defines the model’s behavior, tone, and role.

If you remove it:
The model may respond in a different tone.
It may give longer or less concise answers.
It may not follow the intended instructional style.
Its responses may become inconsistent across sessions.

The system prompt acts as a persistent instruction layer that guides how the assistant behaves throughout the conversation.


3. How does context length affect performance and cost?

Context length refers to the total number of tokens sent to and generated by the model.
As context length increases:

Cost increases because pricing is token-based.
Latency increases because more tokens require more processing.
Memory usage increases on the server.
Performance may degrade if the context approaches the model’s maximum limit.
Long conversations require either:
Trimming old messages, or Summarizing earlier content.

Managing context length is important for scalability and cost efficiency.


4. Why is in-memory storage not production safe?

In-memory storage is not production safe because:

Data is lost if the server restarts.
It does not scale across multiple server instances.
It cannot persist sessions long-term.
It does not support load balancing.
It is vulnerable to memory overflow if many sessions are created.
It is suitable for development and learning, but not for real-world systems.

5. What would change if you used a database?

If a database were used:

Sessions would persist across server restarts.
Multiple server instances could share session data.
The system could scale horizontally.
Old conversations could be archived.
You could implement analytics and monitoring.

The architecture would change to:

Store and retrieve conversation history from a database.
Possibly index by user ID instead of only session ID.
Implement cleanup policies (TTL, expiration).

Using a database makes the system production-ready and scalable.


7. Why does grounding reduce hallucinations?
Grounding reduces hallucinations because the model is forced to base its answer on retrieved, relevant documents instead of relying only on its internal training data. By providing explicit context, the model has concrete evidence to reference, which limits its tendency to generate unsupported or fabricated information.


8. How do chunk size and overlap affect retrieval quality?
Chunk size controls how much information is stored in each embedding:

Small chunks → more precise retrieval but may lose context.
Large chunks → better context but can include irrelevant information.

Overlap helps preserve continuity between chunks:
No overlap → important information may be split across chunks and missed.
With overlap → improves recall by ensuring related content appears in multiple chunks.

The best setup balances precision (small chunks) and context (overlap).


9. What is the difference between semantic search and keyword search?

Keyword search matches exact words or phrases. It fails if different wording is used.
Semantic search uses embeddings to understand meaning and intent, retrieving relevant results even if the exact words differ.

Example:

Query: “function calling itself”
Keyword search might fail if “recursion” isn’t present
Semantic search can still retrieve relevant content about recursion


10. What are common failure modes of RAG systems?

Common issues include:
Poor retrieval: irrelevant chunks are selected
Missing context: important information is not retrieved
Chunking problems: key ideas split across chunks
Over-reliance on LLM: model ignores context and hallucinates
Embedding mismatch: query and document embeddings don’t align well


11. Why are citations important in AI systems?

Citations provide transparency and trust. They allow users to:
Verify where the answer came from
Check correctness of the information
Understand the reasoning behind the response
They also help debug RAG systems by showing whether retrieval worked correctly.
